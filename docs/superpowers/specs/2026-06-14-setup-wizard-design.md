# אשף התקנה מקומי לתכלס

תאריך: 14 ביוני 2026

## מטרה

לאפשר לאדם שאינו מפתח להקים עותק עצמאי של תכלס בחשבונות שלו, בלי למסור
מפתחות לאתר ציבורי ובלי לעבור ידנית על מדריך פריסה ארוך.

האשף ירוץ במחשב המשתמש, יפתח ממשק עברי בדפדפן ויבצע אוטומטית את מרבית
ההתקנה. פעולות שמחייבות מסך הסכמה או הגדרה באתר חיצוני יוצגו כהדרכה
ממוקדת, ולא כאוטומציה מדומה.

## קהל יעד והיקף

- הקהל הראשי הוא משתמש Windows שאינו טכני.
- מפתחים יוכלו לפתוח יומן מתקדם ולראות את הפקודות והפלט.
- הגרסה הראשונה תתמוך ב-Windows בלבד.
- הליבה תופרד מפעולות מערכת ההפעלה כדי לאפשר תמיכה עתידית ב-macOS ו-Linux.
- Telegram יהיה תוסף אופציונלי בסוף ההתקנה, ולא תנאי להפעלת ה-PWA.

## חוויית ההפעלה

המשתמש יפעיל קובץ אחד:

```text
Start-Tachles-Setup.cmd
```

הקובץ יפעיל שרת מקומי על פורט פנוי, ייצור אסימון חד-פעמי ויפתח את האשף
בדפדפן. לא תידרש התקנה של יישום Windows נפרד.

הממשק יהיה אשף משולב:

- בכל מסך תוצג פעולה מרכזית אחת.
- הסבר קצר יופיע בשפה פשוטה.
- אזור מתקפל יציג פרטים טכניים, פקודות ויומן.
- סרגל מצב יציג את ההתקדמות הכוללת.
- ההתקדמות תישמר כדי שאפשר יהיה לסגור ולחזור מאוחר יותר.

## מצבי התקנה

### התקנה בסיסית

מטרתה להביא את המשתמש במהירות ל-PWA עובדת. היא כוללת:

- Supabase ומסד הנתונים.
- Supabase Auth באמצעות קישור כניסה באימייל.
- פריסת ה-PWA ב-Vercel.
- משימות, רשימות, זיכרונות ופרויקטים.
- יצירה וניהול ידניים של תזכורות.
- Push לאחר הגדרת מפתחות VAPID.

יכולות AI יוצעו לבחירה. ללא Anthropic לא תהיה יצירת תזכורות מטקסט חופשי
בעברית, אך מסך התזכורות והיצירה הידנית ימשיכו לעבוד. Google ו-Telegram
יידלגו ויהיה אפשר להוסיף אותם בהפעלה חוזרת של האשף.

### התקנה מלאה

מסומנת כאפשרות המומלצת וכוללת גם:

- Anthropic לניתוב כוונות, תכנון ויכולות Vision.
- OpenAI לתמלול, embeddings ומחקר.
- Google Calendar ו-Google Drive.
- סנכרון Obsidian דרך Drive.
- אפשרות להוסיף Telegram בסיום.

## רצף המסכים

### 1. ברוכים הבאים

- הסבר מה יותקן.
- הבהרה שהעותק נפרס בחשבונות המשתמש.
- הבהרה שהמפתחות נשארים במחשב.
- אישור מפורש לפני התקנה או שינוי של כלי מערכת.

### 2. בחירת סוג התקנה

- התקנה בסיסית.
- התקנה מלאה, מסומנת כמומלצת.
- פירוט קצר אילו יכולות כל אפשרות כוללת.
- אפשרות להרחיב התקנה בסיסית בעתיד בלי להתחיל מחדש.

### 3. הכנת המחשב

בדיקה של:

- Git.
- Node.js.
- npm.
- Deno.
- Supabase CLI.
- Vercel CLI.

האשף יציע התקנה של רכיב חסר רק לאחר אישור המשתמש. אם התקנה אוטומטית אינה
אפשרית, יוצג קישור רשמי והאשף יבדוק שוב לאחר החזרה.

### 4. Supabase

- התחברות באמצעות Supabase CLI.
- בחירה בפרויקט קיים או ניסיון ליצור פרויקט חדש.
- איסוף Project Ref וסיסמת מסד הנתונים.
- החלת migrations.
- יצירת סודות נדרשים.
- פריסת Edge Functions.
- הגדרת Vault עבור המשימות המתוזמנות.
- הגדרת Auth וכתובות Redirect שניתנות לאוטומציה.

אם יצירת פרויקט אינה זמינה דרך הכלים הרשמיים, האשף יפתח את המסך המתאים
ב-Supabase וימשיך לאחר שהמשתמש יזין את פרטי הפרויקט.

### 5. Vercel

- התחברות באמצעות Vercel CLI.
- בחירה בפרויקט קיים או יצירת פרויקט חדש.
- הגדרת Root Directory בתור `apps/web-app`.
- הגדרת משתני `VITE_*`.
- בנייה ופריסה.
- שמירת כתובת ה-PWA.
- עדכון `WEB_APP_URL`, CORS וכתובות Auth לפי הכתובת שנוצרה.

### 6. AI ו-Push

- קליטת מפתחות Anthropic ו-OpenAI בהתאם למצב שנבחר.
- בדיקת המפתחות באמצעות בקשה קטנה שאינה שומרת תוכן משתמש.
- יצירת זוג מפתחות VAPID במחשב.
- העלאת הסודות ל-Supabase.
- הסתרת ערכי המפתחות לאחר שמירתם.

### 7. Google

זהו השלב הידני העיקרי:

- פתיחת Google Cloud במסכים המדויקים.
- הפעלת Calendar API ו-Drive API.
- הגדרת OAuth consent screen.
- הצגת כתובות Redirect מוכנות להעתקה.
- קליטת Client ID ו-Client Secret.
- הגדרת Google Provider ב-Supabase Auth.
- פריסה מחדש של `oauth-callback` והסודות.

האשף לא יבקש הרשאות לחשבון Google מעבר להרשאות שהאפליקציה עצמה צריכה.
הוא ישמור את ההתקדמות לפני פתיחת Google Cloud כדי שהמשתמש יוכל לחזור.

### 8. בדיקות וסיום

האשף יבדוק בפועל:

- כתובת ה-PWA נטענת.
- Supabase Auth מחזיר תצורה תקינה.
- `api-web` נגיש ומגן על נתיבים פרטיים.
- משתמש יכול להיכנס.
- ניתן ליצור משימה וזיכרון.
- ניתן ליצור תזכורת קרובה.
- רישום Push מצליח, אם הופעל.
- Google OAuth מתחיל וחוזר לכתובת הנכונה, אם הוגדר.

בסיום יוצגו:

- כתובת האפליקציה.
- קישור ללוח Supabase.
- קישור לפרויקט Vercel.
- רשימת יכולות פעילות.
- רשימת יכולות שטרם הוגדרו.
- כפתור לפתיחת האפליקציה.

## ארכיטקטורה

### רכיבים

1. **Launcher**

   קובץ CMD קטן שמוודא ש-PowerShell זמין, מפעיל את השרת ופותח את הדפדפן.

2. **Local setup server**

   שרת PowerShell מקומי שמאזין רק ל-`127.0.0.1`. הוא אחראי להרצת פקודות,
   בדיקות, שמירת מצב וכתיבה לקובצי סביבה.

3. **Wizard UI**

   ממשק Web סטטי בעברית. הוא מציג שלבים ויומן, אך אינו מריץ פקודות ישירות
   ואינו מקבל גישה חופשית למערכת הקבצים.

4. **Action modules**

   פעולות מבודדות עבור prerequisites, Supabase, Vercel, secrets, Google
   ובדיקות סיום. כל פעולה תחזיר סטטוס מובנה ותהיה ניתנת להרצה חוזרת.

5. **State store**

   קובץ JSON מקומי ללא סודות. הוא ישמור את מצב השלבים, מזהי הפרויקטים,
   כתובות שנוצרו ותוצאות בדיקה אחרונות.

6. **Secret store**

   `.env.local` ישמש מקור מקומי לסודות הדרושים לסקריפטים הקיימים. הרשאות
   הקובץ יוגבלו למשתמש הנוכחי. ערכי הסודות לא יישמרו ב-state ולא יישלחו
   בחזרה לממשק לאחר השמירה.

## זרימת פעולה

1. הממשק מבקש פעולה מהשרת המקומי.
2. הבקשה כוללת אסימון חד-פעמי שנוצר בהפעלת האשף.
3. השרת מאמת שהבקשה הגיעה מאותו תהליך מקומי.
4. מודול הפעולה בודק מצב קיים לפני שינוי.
5. לפני שינוי רגיש נדרש אישור מפורש מהממשק.
6. הפלט נכתב ליומן לאחר השחרת סודות.
7. בדיקת אימות קובעת אם השלב הושלם.
8. התוצאה וההתקדמות נשמרות ללא סודות.

## אבטחה

- השרת יקשיב רק ל-`127.0.0.1` ועל פורט אקראי.
- בכל הפעלה ייווצר אסימון חדש.
- השרת יגביל Origin לממשק המקומי שפתח.
- API הפעולות יהיה allowlist סגור, ללא endpoint שמריץ פקודת shell חופשית.
- כל ארגומנט לפקודה יועבר כמערך ולא כמחרוזת shell מורכבת.
- סודות יושחרו ביומן ובתשובות HTTP.
- הממשק לא יוכל לקרוא מחדש מפתח מלא שנשמר.
- קובצי סביבה וקובצי מצב יישארו מוחרגים מ-Git.
- קובץ Vault זמני יימחק אוטומטית לאחר שימוש מוצלח, או יוצג למשתמש למחיקה
  אם הפעולה הידנית טרם הושלמה.
- האשף לא יבצע שינוי חיוב או שדרוג תוכנית ללא אישור מפורש.

## טיפול בתקלות והתאוששות

לכל פעולה יהיו המצבים:

- `pending`
- `running`
- `needs_user_action`
- `succeeded`
- `failed`

כישלון יכלול:

- הסבר פשוט בעברית.
- השלב שבו הפעולה נכשלה.
- פעולה מומלצת.
- אפשרות לנסות שוב.
- יומן טכני מתקפל ללא סודות.

פעולות יהיו idempotent ככל האפשר. לדוגמה, פרויקט או secret קיימים יעודכנו
או יאומתו במקום להיווצר שוב. לפני ניסיון חוזר האשף יבדוק את המצב המרוחק.

## שימוש בקוד הקיים

האשף לא ישכפל את לוגיקת הפריסה. הוא יעטוף ויפצל לפי הצורך את:

```text
scripts/deploy-self-host.ps1
```

הפעולות הקיימות להחלת migrations, העלאת secrets ופריסת פונקציות יישארו
מקור האמת. יהיה צורך להפוך אותן למודולים קטנים עם פלט מובנה כדי שהממשק
יוכל להציג התקדמות ולהמשיך לאחר כישלון.
